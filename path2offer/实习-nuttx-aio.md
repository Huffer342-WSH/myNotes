`ltp` 中 `lio_listio()` 相关测试用例触发了 `assert` 报错。最开始的问题表象只是测试失败，但沿着调用链往下看后，我发现问题不在测试本身，而在 `lio_listio()` 原实现的完成通知机制存在并发窗口。

`lio_listio()` 的基本功能，是把一组 `aio_read()` / `aio_write()` 请求打包提交：调用方可以选择 `LIO_WAIT` 等待整组完成，也可以选择 `LIO_NOWAIT` 先返回，并在整组 I/O 全部结束后再收到一次统一通知。

所以这部分工作的主线，其实不是笼统地修改 AIO，而是围绕 `lio_listio()` 机制做一次修复和收敛：一方面修复 LTP 暴露出来的线程安全问题，另一方面把原来比较臃肿的整组回调机制重构成更直接的实现。后续的 `aio_suspend()`、`aio_cancel()`、`work_cancel()`、`aioc` 生命周期调整，本质上都是因为 `lio_listio()` 这条链路被梳理后，相关模块也需要一起修正和补齐。

另外，在定位 `lio_listio()` 问题时，我还发现旧实现为了支持 “整组 I/O 完成后再通知” 这件事，引入了一套依赖信号处理器和 `aio_priv` 的额外机制，代码路径比较绕，状态也不够集中，所以后续不只是修 bug，也顺手做了重构和收敛。

## 核心问题

这阶段我主要处理了以下几类问题：

1. `lio_listio()` 在批量提交 I/O 后再补充整组完成通知上下文，存在明显的线程不安全窗口。
2. `lio_listio()` 的旧版整组回调机制依赖 `SIGPOLL`、`aio_priv` 和额外 signal handler，代码路径较重，状态分散。
3. 为了把 `lio_listio()` 改成“先建立组关系、再提交请求”的模型，需要同时处理“参数非法、实际上没有真正提交”的元素清理问题。
4. 相关配套模块还存在等待、取消、生命周期边界上的问题，会影响 `lio_listio()` 这条链路的整体正确性。

这些问题看起来分散，但主线很明确：核心是 `lio_listio()` 机制本身，其他提交主要是在这个模块完成重构之后，对相关 AIO 路径做配套收敛。

## 分析与定位

### 1. 提交与通知链路中的线程安全问题

`lio_listio()` 的旧实现有一个关键特征：它不是在提交前把“整组 I/O”的完成状态准备好，而是分两步做事。

第一步，`lio_listio()` 先逐个调用 `aio_read()` / `aio_write()` 把请求提交出去。

第二步，如果是 `LIO_NOWAIT` 且传入了 `sig`，再进入 `lio_sigsetup()`：

- 注册一个针对 `SIGPOLL` 的 `lio_sighandler`
- 为每个仍处于 `-EINPROGRESS` 的 `aiocb` 分配一份 `struct lio_sighand_s`
- 把这份私有结构挂到 `aiocbp->aio_priv`

问题就出在这两个阶段之间没有形成可靠的原子关系。`aio_read()` / `aio_write()` 一旦提交成功，worker 线程就可能很快执行并在完成时调用 `aio_signal()`，而旧版 `aio_signal()` 会立即发出 `SIGPOLL`。如果某个 I/O 在 `lio_sigsetup()` 还没来得及给它写入 `aio_priv` 之前就完成了，那么后续 `lio_sighandler()` 里执行下面这段逻辑时：

- 先从 `info->si_value.sival_ptr` 取回 `aiocbp`
- 再从 `aiocbp->aio_priv` 取回 `struct lio_sighand_s *`
- 然后 `DEBUGASSERT(sighand && sighand->list)`

这里就可能拿到 `NULL`，直接触发断言。也就是说，LTP 暴露出来的 assert，本质上就是 “I/O 已完成并触发信号，但 `lio_listio()` 还没来得及建立整组完成通知上下文”。

如果用更直接的时序来描述，就是：

1. `lio_listio()` 提交第一个 `aio_read()` / `aio_write()`
2. worker 线程迅速完成该请求并调用 `aio_signal()`
3. `aio_signal()` 发送 `SIGPOLL`
4. `lio_sighandler()` 被调度执行
5. 此时 `lio_sigsetup()` 还没给这个 `aiocb` 填好 `aio_priv`
6. `lio_sighandler()` 取到空指针并命中断言

这就是旧实现线程不安全的核心原因：整组通知所依赖的私有状态是在 I/O 提交之后才补写的，而完成事件可能先发生。

我在分析时把关注点放在“单个 I/O 完成”和“整组 I/O 完成”这两个语义是否共享同一套可靠状态。原实现里，整组状态依附在提交后的补写动作上，这就是根因。换句话说，旧实现把“建立状态”和“消费状态”放在了两个并发线程里，但没有保证前者一定先于后者发生。

### 2. 旧实现为什么会采用“先提交，再建立整组关系”

进一步读旧代码后，我认为原作者这样设计并不是随意写出来的，而是有一个现实考虑：`aio_read()` / `aio_write()` 在真正提交到异步执行前，会先做参数检查。像无效文件描述符、非法参数这类情况，函数会直接失败返回，根本不会进入真正的异步执行流程。

从这个角度看，旧实现想避免把这些“根本没提交成功”的元素也纳入整组完成通知，所以才选择：

- 先调用 `aio_read()` / `aio_write()`
- 再对仍处于 `-EINPROGRESS` 的 `aiocb` 建立整组通知关系

这个出发点本身是合理的，问题在于它把“筛掉未提交元素”和“避免并发窗口”这两个目标放在了同一套后置处理逻辑里，最终为了处理前者，牺牲了后者。

### 3. 新实现为什么仍然要在提交后清理链表元素

我在重构时没有简单地把所有元素永久挂进链表，而是保留了对“未真正提交元素”的剔除逻辑。

新的做法是：

- 先把所有非 `LIO_NOP` 元素挂入链表，先建立整组关系，消除竞态窗口
- 再逐个调用 `aio_read()` / `aio_write()`
- 提交后检查返回值以及 `aio_result`
- 对于没有真正进入异步执行的元素，例如直接返回错误、或立即暴露出 `-EBADF` / `-EINVAL` 的情况，再把它们从链表中删除

这样做其实是同时兼顾了两件事：

- 保证整组状态在任何 I/O 完成之前就已经存在，不会再出现旧版 `aio_priv` 的竞态
- 保留旧实现中“不要把未真正提交的请求算进整组完成条件”这个合理语义

也就是说，新版不是否定原作者对参数校验场景的考虑，而是把这个考虑放到了一个线程安全的实现框架里。

### 4. 整组回调还有一个细节：要处理“链表已经空了”的场景

整组通知机制还有一个容易忽略的细节：当 `LIO_NOWAIT` 且传入 `sig` 时，`lio_listio()` 最后需要决定，是等待后续 `aio_signal()` 在最后一个节点完成时发送整组通知，还是当前就直接补发通知。

为此我在实现里会检查链表是否已经为空。

这个判断覆盖了两类情况：

- 所有元素都因为参数错误等原因没有真正提交，已经在提交后被从链表中删掉
- 某些请求完成得非常快，在 `lio_listio()` 返回前链表就已经被 `aio_signal()` 清空

如果此时链表已空，就说明后续不会再有“最后一个节点”替我们触发整组通知，因此需要在 `lio_listio()` 末尾主动执行一次 `nxsig_notification()`。这个细节保证了整组通知语义完整，不会因为完成过快或未真正入队而漏发回调。

### 5. 等待与唤醒语义不严格

`aio_suspend()` 的语义应该是“等待传入 `list` 中任意一个 I/O 完成”，但实现上只要收到了 `SIGPOLL` 就可能返回。这样一来，只要进程内存在其他无关 AIO 完成，也可能把当前等待提前唤醒，和接口语义不一致。

这个问题的关键不在信号本身，而在“收到信号之后有没有重新核验目标条件”。如果没有针对 `list` 做二次检查，就会把“任意 AIO 完成”误当成“目标 AIO 完成”。

### 6. 取消路径与 work queue 运行态边界处理不完整

`aio_cancel()` 既要遍历 pending 队列，又要和底层 `work_cancel()` 的返回语义配合。当目标工作项已经开始执行时，`work_cancel()` 应返回“队列中已不存在该工作项”的语义；但如果上层取消逻辑没有在失败后正确推进遍历指针，就会在持锁状态下反复处理同一个正在运行的请求，形成死循环。

这里我关注的是“队列态”和“运行态”的边界。当一个请求已经不在队列中，它就不应继续按照“可取消队列项”处理。

### 7. 生命周期释放时机过早

`aio_read()` / `aio_write()` / `aio_fsync()` 的 worker 原来会先 `aioc_decant()`，再执行实际 I/O。这样虽然能尽早释放容器，但会让后续 I/O 过程继续依赖一个已经脱离容器保护的 `aiocb` 相关状态。一旦调用方过早关闭文件或复用对象，就可能触发 use-after-free 或产生读写错误。

这个问题本质上是典型的生命周期管理错误：为了减少容器占用而提前释放，但释放动作跨过了“实际 I/O 已结束”这一安全边界。

## 方案设计

### 1. 重构 `lio_listio()`，把整组完成状态前置为显式链表

我把 `lio_listio()` 的实现从 `libs/libc/aio` 调整到 `fs/aio` 侧，并把“整组请求”的完成状态显式建模为 `aiocb` 链表：

- 在提交前就把所有非 `LIO_NOP` 的 `aiocb` 挂入同一条链表
- 在 `aiocb` 中增加 `lio_link`、`lio_sigevent`、`lio_sigwork` 等关联字段
- 每个单项 I/O 完成时，在 `aio_signal()` 中把自己从链表移除
- 只有链表清空时，才触发整组 `lio_listio()` 的完成通知
- 对于提交失败、未真正进入异步执行的元素，在提交后再从链表删除

这样做的核心收益是把“整组完成”的判定从脆弱的后置补写，改成稳定的共享状态，避免了完成回调早于状态初始化的问题。

和旧实现相比，新的做法有两个本质变化：

- 不再依赖 `SIGPOLL + signal handler + aio_priv` 这套间接机制去回收整组状态
- 不再要求“先发完成信号，后补充私有上下文”，而是在提交前就把整组关系建立好，再在提交后剔除未真正入队的元素

因此旧版的竞态窗口被直接消掉了，代码结构也明显比 `lio_sighand_s` + `lio_sigsetup()` 的方式更直接。

另外，我在 `lio_listio()` 末尾增加了“链表是否已经为空”的检查。如果链表此时为空，就主动补发整组完成通知，避免因为请求完成过快或根本没有真正入队而漏掉回调。

### 2. 配套修正 `aio_suspend()` 的等待逻辑

我把 `aio_suspend()` 改成循环检查模式：

- 先遍历 `list`，确认目标 I/O 是否已经完成
- 若未完成，再等待 `SIGPOLL`
- 被唤醒后重新检查 `list`
- 若设置了超时，则基于 `CLOCK_MONOTONIC` 重新计算剩余时间

这样即使有无关 I/O 发出 `SIGPOLL`，也不会错误返回。返回条件重新绑定到“目标 `list` 中存在完成项”这个正确语义上。

这部分不是独立需求，而是 `lio_listio()` 这条链路梳理完之后，对等待语义做的配套修正。

### 3. 让取消逻辑和 `work_cancel()` 的返回语义对齐

我补充了 `work_cancel()` 在“无此排队工作项”场景下返回 `-ENOENT` 的语义，然后在 `aio_cancel()` 中据此修正遍历行为：

- 遍历 pending 队列时先保存 `next`
- 如果当前 `aioc` 不匹配目标 `fildes`，直接跳过
- 如果 `work_cancel()` 失败，说明请求可能已经进入运行态，不再反复处理当前项
- 只有取消成功时才 decant、标记 `ECANCELED` 并发送信号

这让取消流程能正确区分“仍在队列中，可取消”和“已经开始运行，不可取消”两种状态。

这部分也是 `lio_listio()` 相关路径的配套修改，因为批量提交后的取消行为同样依赖底层工作项状态边界。

### 4. 把 `aioc_decant()` 推迟到 I/O 真正完成之后

在 `aio_read()`、`aio_write()`、`aio_fsync()` 的 worker 中，我把 `aioc_decant()` 从 I/O 之前移到 `aio_signal()` 之后：

- worker 先从容器中读取必要上下文
- 保持容器和文件引用在整个 I/O 期间有效
- I/O 结束后更新 `aio_result`
- 完成通知发送之后再 decant 和释放容器

这样既避免了 use-after-free，也避免了调用方过早关闭文件导致的异步读写异常。

  价值：在保留原有参数校验语义的前提下，消除了旧版 “先提交、后补状态” 的线程不安全窗口，同时简化了整组回调机制。

- `1e072f8ed0`
  修复点：调整 `aio_suspend()`，从“收到信号即返回”改为“收到信号后重新核验目标 list”。
  价值：作为 `lio_listio()` 相关等待路径的配套修正，让接口行为重新符合 POSIX 语义，避免被无关 I/O 错误唤醒。

- `d0ad0928f8`
  修复点：修正 `work_cancel()` 在工作项不存在时的返回值语义。
  价值：为 `lio_listio()` 相关取消路径判断“已开始执行还是仍在队列中”提供了可靠依据。

- `4380539d44`
  修复点：修复 `aio_cancel()` 在取消运行中任务时可能进入死循环的问题。
  价值：补齐了 `lio_listio()` 批量提交后的取消路径和 work queue 运行态边界处理，避免持锁死循环。

- `337fc91682`
  修复点：把 `aioc` 释放时机后移，修复 use-after-free 和过早关闭文件带来的风险。
  价值：完善了 `lio_listio()` 相关异步请求的生命周期管理，保证 I/O 期间相关对象持续有效。

## 收获总结

这条主线让我比较系统地锻炼了以下能力：

- 并发问题定位能力：能从 LTP 测试中的 assert 反推出 `lio_listio()` 完成通知机制里的真实竞态窗口。
- 接口语义理解能力：能理解旧实现为什么要区分“真正提交成功的元素”和“参数检查阶段就失败的元素”，并在新实现里保留这个语义。
- 机制重构能力：不是只修一个断言，而是把整组回调机制从 `SIGPOLL + aio_priv + signal handler` 收敛成更直接的链表模型。
- 工程化收敛能力：以 `lio_listio()` 为主线，把等待、取消和生命周期等相关路径一起补齐。
